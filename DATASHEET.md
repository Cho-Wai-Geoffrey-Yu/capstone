# Datasheet — BBO Capstone Query/Response Dataset

Following the template proposed by Gebru et al. (2018), *Datasheets for
Datasets*.

## Motivation

**For what purpose was the dataset created?**
The dataset consists of the input query points submitted to 8 unknown
black-box functions over 22 rounds of a Bayesian Optimization capstone
project, together with the scalar outputs returned by the course platform.
It was created to (a) drive the sequential optimisation process itself, and
(b) provide a transparent, reproducible record of what was queried and why.

**Who created the dataset and on behalf of which entity?**
Created by the author (a postgraduate ML programme participant) as part of a
structured capstone assignment. The underlying black-box functions themselves
were created and are owned by the course platform (Emeritus); their exact
definitions were never disclosed to participants.

**Who funded the creation of the dataset?**
No external funding; created as coursework.

## Composition

**What do the instances represent?**
Each instance is one (input, output) pair: an input vector `x ∈ [0,1]^d`
(d = 2, 2, 3, 4, 4, 5, 6, 8 for Functions 1–8 respectively) and the
corresponding scalar output `y = f(x)` returned by the platform.

**How many instances are there in total?**
22 instances per function (10 provided as an initial seed dataset, 12
submitted by the author across Modules 12–23), i.e. 176 (input, output) pairs
in total across all 8 functions.

**Does the dataset contain all possible instances, or is it a sample?**
It is an extremely sparse, non-random sample. For an 8-dimensional function,
22 points represent a vanishingly small fraction of the continuous input
space — sample density decreases sharply with function dimensionality.

**What data does each instance consist of?**
- `x`: a vector of floats in [0,1], six decimal places, one value per
  dimension.
- `y`: a single float (unbounded — observed values in this project ranged
  from roughly -40 to +8,662 depending on the function).

**Is there a label or target associated with each instance?**
The output `y` functions as the optimisation target (to be maximised).

**Is any information missing from individual instances?**
No — every submitted query received exactly one corresponding output.

**Are relationships between instances made explicit?**
Yes — instances are ordered sequentially by query round, and later queries
were chosen using a GP model conditioned on all earlier instances for that
function. This sequential dependency is central to interpreting the data:
later points are *not* independent samples, they are adaptively chosen based
on everything observed before them.

**Are there recommended data splits?**
Not applicable — this is not a train/test prediction dataset; it is a
sequential decision-making trace. Standard train/test splitting would misuse
the data, since later points depend on earlier ones by design (see
"Uses" below for further caveats).

**Are there errors, noise, or redundancies?**
GP models fit to the data show non-negligible observation noise for some
functions (e.g. Function 2 required a higher noise regularisation term,
α=1e-4, versus α=1e-8 for cleaner functions such as Function 5), suggesting
either genuine stochasticity in some functions or floating-point/formatting
noise. This is not independently verifiable since ground-truth functions are
unknown.

**Is the dataset self-contained, or does it rely on external resources?**
Self-contained. All 176 (x, y) pairs used in this project's final analysis
are stored in `data/final_results_summary.json`.

## Collection Process

**How was the data acquired?**
Each `y` value was obtained by submitting an `x` vector to the capstone
project portal (a web form provided by the course), which returned the
corresponding score. All `x` vectors were generated by the author's
optimisation code (`src/bbo_optimizer.py`), not sampled from an external
source.

**What mechanisms were used for data collection?**
Manual submission through a web portal, with `x` vectors computer-generated
via Gaussian Process–based acquisition optimisation.

**Over what timeframe was data collected?**
Approximately 12 weeks, one round per week (Modules 12 through 23/24 of the
programme).

## Preprocessing / Cleaning / Labelling

No preprocessing was applied to the raw `(x, y)` pairs beyond formatting `x`
to six decimal places as required by the submission portal. No cleaning,
filtering, or labelling was performed — all submitted queries and their
returned outputs are retained in full.

## Uses

**Has the dataset been used for any tasks already?**
Yes — exclusively for this capstone's sequential Bayesian Optimization task.

**What (other) tasks could the dataset be used for?**
Limited reuse value beyond this project: because points were chosen
adaptively and adversarially toward each function's discovered optimum, the
dataset is heavily biased toward high-performing regions (see `MODEL_CARD.md`
§ Limitations & Biases) and is unsuitable as a general-purpose benchmark or
for training a global surrogate model intended to generalise across the full
input domain.

**Are there tasks for which the dataset should not be used?**
It should not be used to draw conclusions about the *global* shape of any
function (e.g. "Function 5 is monotonically increasing toward its
boundary") — the sampling is concentrated and adaptive, not representative.
It should not be treated as i.i.d. data for standard supervised-learning
train/test evaluation, for the sequential-dependency reasons noted above.

## Distribution

**Will the dataset be distributed to third parties?**
The (x, y) pairs generated by the author are shared openly in this public
repository. The underlying black-box functions are **not** redistributed
(they are not owned by the author and were never disclosed).

**How will the dataset be distributed?**
Via this public GitHub repository, in `data/final_results_summary.json`
(consolidated summary) — see repository README for access instructions.

## Maintenance

**Who maintains the dataset?**
The author. As this is a completed, static coursework artefact, no ongoing
updates are planned; the dataset represents a fixed historical record.
